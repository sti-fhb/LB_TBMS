# ET 模組建構概念

本文件說明**教育訓練模組（ET）**的需求背景、UseCase 設計脈絡、平台選型理由（Moodle LMS），以及後續 SD（System Design / 系統設計）階段的進行建議。目的：讓接手的開發 / 設計人員能快速理解「為什麼這樣建」，而不只是「要建什麼」。

---

## 1. 模組定位與目標

### 1.1 模組在系統中的角色

ET 模組是 TSBMS 的**周邊支援系統**，不直接參與血液業務主流程（採血 / 成分 / 檢驗 / 供應 / 醫務 / 標籤），而是解決**人員能力養成**與**作業知識傳承**的問題。

| 主系統 | ET 模組 |
|--------|---------|
| 處理血品（物流、資料） | 處理人員（訓練、考核、文件） |
| 資料整合、流程管控 | 知識整合、能力管控 |
| 實時作業 | 非同步學習 |

### 1.2 核心目標

1. **跨站點人員能力標準化**：三總與國軍醫院多個站點，透過統一線上課程確保 SOP 認知一致。
2. **作業規範文件管理**：SOP 文件版本化、集中存取，避免各站點版本不一致。
3. **操作即時協助**：使用者在任何畫面按 [F1] 可查閱對應功能的操作手冊。
4. **訓練歷程可追溯**：管理者可掌握各人員完課率、測驗成績，作為人員資格認證參考。

## 2. 需求來源與 UseCase 設計脈絡

### 2.1 RFP 需求來源

ET 模組的需求來自 RFP 章節 **1.7 文件管理與教育訓練**，對應 RQDP041~043（含 RQDP044 中文化介面）。依「RFP 章節決定 RQ 編碼」原則，這些需求保留 **RQDP** 前綴（基礎架構章節），但實作歸屬 ET 模組。

> 追溯鏈：UCETxxx → RQDP041~043 → RFP 1.7 文件管理與教育訓練

本模組另有 `_refs/10-教育訓練.md` 分析資料作為實作細節來源，已逐條整理為 RQET001~008（分析需求）。

### 2.2 角色分析

ET 模組涉及四個角色，功能需求沿角色分流：

| 角色 | 職責 | 對應 UseCase |
|------|------|--------------|
| **管理者（講師）** | 建課、編輯教材、建測驗、檢視學習報表 | UCET001~006 |
| **學員** | 加入課程、觀看內容、測驗、查詢個人進度 | UCET007~010 |
| **資訊人員** | 維護 SOP 文件、上傳操作手冊至指定位置 | UCET011 |
| **使用者**（TSBMS 一般操作者） | 按 [F1] 查閱線上操作手冊 | UCET012 |

### 2.3 UseCase 結構設計原則

**（1）管理者與學員流程對偶設計**

管理者的「建立」行為對應學員的「使用」行為，兩側流程成對出現：

| 管理者端 | 學員端 | 共同 Moodle 功能 |
|---------|--------|----------------|
| UCET001 建立課程 | UCET007 加入課程 | ET01, ET09, ET10, ET19 |
| UCET002 上傳教材 → UCET003 編排章節 | UCET008 學習課程內容 | ET02, ET03, ET04, ET05, ET06, ET14 |
| UCET004 建立測驗 | UCET009 進行線上測驗 | ET07, ET08, ET15 |
| UCET006 檢視學習報表 | UCET010 查看個人學習進度 | ET11, ET12, ET13, ET16 |

**（2）文件管理拆分為兩條獨立 UC**

為避免把「文件維護」與「文件查閱」混在同一個 UC，刻意拆成：

- **UCET011 維護 SOP 文件與版本**：資訊人員端的寫入流程（對應 Moodle Wiki 模組 ET17）
- **UCET012 查閱線上操作手冊**：使用者端 F1 觸發的讀取流程（對應 Moodle Book 模組 ET18）

兩者整合進 Moodle 的不同模組，也對應不同實作策略（Wiki 支援協作編輯；Book 適合多章節結構化閱讀）。

## 3. 為什麼採用 Moodle LMS

### 3.1 選型考量面向

| 面向 | Moodle 表現 |
|------|------------|
| 授權成本 | 開源免費（GPL），無授權費用 |
| 功能完整度 | 課程、章節、測驗、作業、Wiki、Book、報表、通知一應俱全 |
| 政府 / 軍方採用紀錄 | 台灣多所大學、國防大學、部分醫療機構皆採 Moodle |
| 中文化 | 原生支援繁體中文介面與多語系 |
| 擴充性 | Plugin 架構成熟（目前已規劃 enrol_invitation 擴充邀請信功能） |
| 社群活躍度 | 全球超過十萬個 Moodle 站點，社群與文件豐富 |
| 維運負擔 | LAMP 技術棧（PHP + MySQL/Postgres），導入資源容易找 |

### 3.2 與其他方案比較

| 方案 | 成本 | 功能 | 客製彈性 | 導入時間 | 結論 |
|------|------|------|---------|---------|------|
| **Moodle** | 免費 | 高 | 高（Plugin） | 中 | ✓ **採用** |
| 自建 LMS | 高（開發） | 可控 | 極高 | 長 | ✗ 非本案核心，不值得自建 |
| Canvas / Blackboard | 高（授權） | 高 | 中 | 短 | ✗ 授權費與採購流程 |
| Google Classroom | 低 | 低 | 低 | 短 | ✗ 資料落在境外且無 SOP 版本管理 |

### 3.3 採用 Moodle 的約束條件

本案將 Moodle 視為**獨立子系統**，以「原生功能優先、最小客製」為原則：

1. **不改 Moodle 核心程式**，僅透過 Plugin（如 `enrol_invitation`）擴充。
2. **課程結構依 Moodle 規範設計**（Course → Section → Activity/Resource），不自行抽象資料模型。
3. **使用者身分**：與 TSBMS 做 **SSO 整合**（SAML / OAuth），不在 Moodle 內重複維護帳號。
4. **資料庫獨立**：Moodle 使用自有 DB（MySQL/Postgres），TSBMS 不直接讀寫 Moodle DB。

## 4. 與 TSBMS 主系統的整合邊界

```
┌─────────────────────────────────────────┐      ┌──────────────────────┐
│           TSBMS 主系統                   │      │   Moodle LMS          │
│  ┌──────────────────────────────────┐   │      │                       │
│  │ 血液業務作業（BC/CP/TL/BS/MA/LB）│   │      │  ┌─────────────┐     │
│  └──────────────────────────────────┘   │      │  │ 課程 / 測驗  │     │
│                                          │      │  │ Wiki / Book │     │
│  ┌──────────────────────────────────┐   │SSO   │  └─────────────┘     │
│  │ 使用者登入 / 角色管理（DP）      │◄──┼──────┤                       │
│  └──────────────────────────────────┘   │      │  ┌─────────────┐     │
│                                          │      │  │ 學習報表     │     │
│  ┌──────────────────────────────────┐   │ F1   │  └─────────────┘     │
│  │ 操作畫面（各模組）               │───┼─────►│                       │
│  │   按 F1 → func_name              │   │ URL  │  ┌─────────────┐     │
│  └──────────────────────────────────┘   │      │  │ Book 模組    │     │
│                                          │      │  │ （操作手冊） │     │
└─────────────────────────────────────────┘      │  └─────────────┘     │
                                                  └──────────────────────┘
```

### 4.1 三個整合接點

| 接點 | 方向 | 技術 | 對應 UC |
|------|------|------|---------|
| **SSO 身分驗證** | TSBMS → Moodle | SAML / OAuth | 所有需登入的 UC |
| **操作手冊 F1 連結** | TSBMS → Moodle（URL） | HTTP 跳轉 + func_name 對應 | UCET012 |
| **邀請信通知** | Moodle → 學員 | SMTP（Moodle Plugin） | UCET005 |

### 4.2 明確不做的事

- ❌ 不把 Moodle 資料同步回 TSBMS DB
- ❌ 不在 TSBMS 內重建學習功能
- ❌ 不客製 Moodle 核心（只允許 Plugin 擴充）
- ❌ 不用 TSBMS 的捐血者資料做 Moodle 學員基礎

## 5. SD 階段進行建議

### 5.1 建議的進行順序

```
階段 1：平台建置
  └─ Moodle 安裝（Docker Compose）、基礎設定、中文化、主題

階段 2：SSO 整合
  └─ 與 TSBMS DP 模組（帳號管理）的身分驗證整合

階段 3：內容架構規劃
  ├─ 課程分類設計（採血SOP / 成分SOP / 檢驗SOP / 供應SOP / ...）
  ├─ Book 模組操作手冊結構（一個功能一本 Book，命名對應 func_name）
  └─ Wiki 模組 SOP 文件結構

階段 4：邀請信 Plugin 客製
  └─ enrol_invitation Plugin 整合、郵件模板調整

階段 5：F1 整合
  ├─ TSBMS 各畫面加上 F1 handler
  └─ func_name ↔ Moodle Book URL 的對應表

階段 6：測試 / 上線
  ├─ 建立示範課程、測試學員流程
  ├─ 驗收完課率、測驗成績報表
  └─ 文件人員培訓（如何使用 Wiki 編輯 SOP）
```

### 5.2 SD 決策檢核清單

各功能 SPEC 撰寫時（`functions/ETxx-*.md`），建議至少釐清以下：

| 項目 | 為什麼重要 |
|------|-----------|
| Moodle 原生 vs 需客製 | 決定導入成本與時程 |
| 學員是否需要 TSBMS 帳號 | 影響 SSO 設計 |
| 資料儲存在 Moodle DB 還是要外掛 | 影響維運備份 |
| 檔案（影片 / PDF）存哪裡 | Moodle 預設存磁碟，大檔案可能需要 S3 / NAS |
| 權限對應關係 | Moodle Admin / Teacher / Student 對應 TSBMS 哪些角色 |
| 通知管道 | Moodle 原生 Email vs 整合 TSBMS 通知系統 |

### 5.3 建議的技術選型

| 面向 | 建議 |
|------|------|
| Moodle 版本 | 4.x LTS（長期支援版） |
| 部署方式 | Docker Compose（開發） / Kubernetes（生產） |
| DB | PostgreSQL（與 TSBMS 主系統一致，降低維運技能要求） |
| 檔案儲存 | 本地磁碟起步，擴充時評估 S3 相容儲存 |
| SSO 協定 | SAML 2.0（Moodle auth_saml2 Plugin） |
| SMTP | 與 TSBMS 共用郵件服務 |

### 5.4 常見風險與因應

| 風險 | 因應 |
|------|------|
| Moodle 升級破壞客製 Plugin | 僅用官方認證的 Plugin，版本升級前測試 |
| 影片儲存空間快速成長 | 初期制訂影片上傳規範（解析度 / 長度上限），留彈性遷移到物件儲存 |
| SOP 文件格式各異 | 統一要求 Wiki / Book 格式（Markdown / HTML），禁止只放 Word 附件 |
| F1 操作手冊缺漏 | 每個新功能上線必要件：對應 Book 文件 |
