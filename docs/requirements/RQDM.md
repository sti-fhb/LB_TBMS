# 文件管理模組（DM）需求文件

> **模組代碼**：DM（Document Management，文件管理）
>
> **定位**：獨立於 TSBMS 主系統部署之文件管理服務，採主系統相同技術棧（FastAPI + React）開發，但自成一服務。透過 **SS 模組**（獨立認證系統，與主系統 DP 完全切開）提供之認證 API 取得 Token 後運作；系統參數（如 `DM_UPLOAD_PATH`）仍透過 DP **APIDP003** 取得。
>
> **2026-04-28 變更**：使用者認證由 DP API 改為 **SS API（APISS001 / APISS002）**；DM 使用者為一般軍人，由 SS 模組獨立管理。
>
> **歷史**：原為獨立模組 → 2026-03-25 併入 ET（教育訓練） → 2026-04-24 因採自建路線、不走 Moodle Wiki/Book，重新獨立為 DM 模組。

## 1. 模組概述

### 1.1 角色

| 角色 | 職責範圍 |
|------|----------|
| 使用者（任一 SS 角色 + DM01/DM05 功能權限）| 上傳文件、建立審查單、加入文件明細、送審、讀取 |
| 審查者（SOP_REVIEWER）| 對審查單作核准 / 退件（**唯一硬卡點角色**）|
| 系統自動 | 版本歷程維護、通知發送、稽核軌跡記錄 |

> 角色簡化（2026-04-27 / 2026-04-28 更新）：只硬卡 `SOP_REVIEWER`，其餘任一 **SS 角色** 視為一般使用者，是否能進入 DM 各功能由 DM01 / DM02 / DM05 等功能選單授權決定。原 SOP_AUTHOR / SOP_READER 名稱廢除。SS 角色於 SS 模組獨立管理（詳見 [RQSS.md](RQSS.md)）。

### 1.2 作業流程

```
文件撰稿與發布
──────────────
1. 資訊人員登入 DM（透過 **SS API（APISS001）** 認證取得 Token + 角色 + 可用功能）
2. 於文件管理頁面選擇分類（**SOP 或 操作手冊**，DM 僅管理此兩類）
3. 上傳 PDF/Word 或使用線上編輯器撰寫
4. 填寫版本號（v1.0 / v2.0）、變更摘要
5. 送審者建立審查單、加入文件明細、送審 → 系統通知具 SOP_REVIEWER 角色之審查者（per UCDM006 文件審查作業流程）
6. 審核者檢視、核准或退件（帶回意見）
7. 核准後系統保留舊版為歷程、發布新版、通知訂閱者

查閱
────
8. 使用者於主系統功能畫面觸發線上操作手冊 → 主系統 call DM API（帶 func_name）→ DM 回應對應手冊頁面
9. 使用者於 ET 模組教材內點 SOP 連結 → 轉跳 DM 文件頁瀏覽
10. 使用者於 DM 首頁 / 分類頁搜尋文件、瀏覽版本歷程
```

### 1.3 整合邊界

| 接點 | 方向 | 用途 |
|------|------|------|
| **SS API（認證）** | DM → SS | DM 以 SS 帳密換取 Token + 角色 + 可用功能（APISS001）|
| **SS API（Token 驗證）** | DM → SS | 後續每次請求驗 Token（APISS002）|
| DP API（系統參數）| DM → DP | 取系統參數（`DM_UPLOAD_PATH` 等，APIDP003 + 內建靜態 Token）|
| DP API（Audit Log）| DM → DP | 寫共用稽核軌跡（DM 自身審查軌跡寫於 DM_AUDIT，跨系統重大事件可同步寫 DP_AUDIT_LOG）|
| 主系統功能畫面 → DM | 主系統 → DM | 開啟線上操作手冊（by func_name）|
| ET 模組教材連結 → DM | ET 模組 → DM | 教育訓練教材中連結至 SOP / 手冊查閱 |
| DM 通知 → Email / 主系統 | DM → 外部 | 文件發布 / 簽核通知 |

### 1.4 可配置參數

- 文件分類代碼（**SOP / 操作手冊 / 訓練教材** 共三類；訓練教材限文件型，影片仍由 ET 模組自管；表單不在 DM 範圍）
- 各分類對應的儲存路徑（base path / bucket / 子目錄規則）
- 簽核流程規則（單層 / 會簽）
- 版本號規則（語意 / 流水）
- 檔案格式限制、單檔大小上限
- 通知範本

### 1.5 對應功能選項

| 功能選項 | 用途 | 對應 UC |
|---------|------|---------|
| DM01-文件管理作業 | 文件上傳、分類標記、版本管控、送審 / 審核 | UCDM001（亦為 UCDM002 上傳前置作業入口）|

| RQDM007 | DM 服務同 LB（LBSB01）模式，於程式內以**常數**設定一組對外存取 Token（DP_PARAM_D `PARAM_ID='DP_EXT_API_KEY'`、`PARAM_KEY='DM'`、`PARAM_VALUE=<INTERNAL_STATIC_TOKEN>` 永不過期），用於呼叫主專案 **APIDP003-系統參數查詢** 取得 `DM_UPLOAD_PATH` 等啟動參數；不沿用 APIDP001 動態換 Token 流程 | 系統自動 | 分析資料 |
| RQDM008 | **檔名作為文件邏輯識別**：`(CATEGORY, BASE_FILENAME)` 為 DM_DOC 唯一識別。同分類同檔名再次上傳一律視為**同一份文件的新版**（不開放「另存為新文件」）；若使用者意在建立另一份同名文件，須改檔名後再上傳。檔名標準化：空格→底線、Unicode NFC、移除前後空白；BASE_FILENAME 大小寫不敏感；副檔名一致才視為同檔 | 資訊人員 | 分析資料 |
| RQDM009 | **版本號規則與唯一性**：預設格式 `V{MAJOR}.{MINOR}` 或 `V{MAJOR}.{MINOR}R{REVISION}`；第一版固定 `V1.0`；既有文件再上傳預填建議「最低衝擊」版號 `Vx.yR(z+1)`，使用者可手改但不可與該文件**任何先前已發布版本**重複（退件版本不占用版號）。流水版（rev 1/2/3）由分類層級配置 | 資訊人員 | 分析資料 |
| RQDM010 | **最新版檔名固定無版本後綴**（外部連結穩定）：最新發布版實體檔案放 `{原目錄}/{BASE_FILENAME}`（無版本後綴）；歷史版本檔案統一移到 `{原目錄}/歷史版本/{BASE_FILENAME 含版本後綴}`，例如 `LBSB01_操作手冊V1.0.pdf`。歷史版本目錄不開放外部 URL，下載僅能透過 DM 後端轉發（呼應 NFR-006） | 系統自動 | 分析資料 |
| RQDM011 | **上傳新版的原子搬移流程**（簽核通過後執行）：(A) 鎖定 DOC_ID；(B) 重命名舊最新檔加上版號後綴；(C) 移到 {原目錄}/歷史版本/；(D) 新檔寫入 {原目錄}/{BASE_FILENAME}；(E) 同 transaction 更新 DM_DOC_VERSION（舊版 IS_CURRENT=0、新版 IS_CURRENT=1）與 DM_DOC.CURRENT_VERSION_ID + DM_AUDIT；(F) 釋放鎖。失敗須完整回滾檔案與 DB | 系統自動 | 分析資料 |
| — | 文件依分類（**SOP / 操作手冊 / 訓練教材**）儲存，支援標籤與全文檢索 | 使用者 | 分析資料 |
| — | 搜尋結果列表呈現文件名、分類、版本、發布日期、作者 | 使用者 | 分析資料 |

## 8. 跨模組介接

| 呼叫方 | 目標 | 方式 | 用途 |
|--------|------|------|------|
| DM | **SS（APISS001）** | HTTP（IP 白名單 / mTLS） | 使用者登入認證，取 Token + 角色 + 可用功能 |
| DM | **SS（APISS002）** | HTTP（IP 白名單 / mTLS） | 每次請求 Token 驗證 |
| DM | DP（**APIDP003**）| HTTP + 內建 Token | 取系統參數（`DM_UPLOAD_PATH` 等），決定檔案落地路徑 |
| DM | DP（Audit Log API）| HTTP + 內建 Token | 跨系統重大事件寫共用稽核軌跡（DM 自身審查軌跡寫 DM_AUDIT）|
| 主系統各模組 | DM（API）| HTTP + Token | 線上操作手冊查閱（by func_name）|
| ET 模組 | DM（URL 轉跳）| HTTP Redirect | 教材內 SOP / 手冊超連結 |
| DM | 主系統通知中心 / Email | API / SMTP | 文件發布通知 |

> **DM 內建 Token**：DM 程式內以常數寫入 Token（同 LBSB01 模式），對應 DP 端 `DP_PARAM_D` PARAM_ID='DP_EXT_API_KEY'、PARAM_KEY='DM' 的 PARAM_VALUE，永不過期。DM 啟動時即可呼叫 APIDP003 取系統參數，不需先走 APIDP001 換 Token。
>
> **與 SS 認證之區分**：DM 內建 Token **僅供呼叫 DP（APIDP003 系統參數、Audit Log）**，**不**用於使用者認證；使用者認證一律走 SS 之 APISS001 / APISS002 動態 Token 流程。
