# 提示詞參考

本文件收錄協作時使用的標準提示詞，確保所有同仁產出格式一致。

## 整體產出順序

整體流程參考 CLAUDE.md 的「需求整理流程」與「Speckit 作業流程」。

## Speckit 規格產出

### 作業順序總覽

| 順序 | 做什麼 | 怎麼跟 Claude 說 | 產出 |
|------|--------|------------------|------|
| 1 | 產出功能規格 | 見 Step 1 | spec.md |
| 2 | 釐清規格 | 見 Step 2（可多輪） | 更新 spec.md |
| 3 | 產出實作計畫 | 見 Step 3 | plan.md, research.md, data-model.md, contracts/ |
| 4 | 產出開發任務 | 見 Step 4 | tasks.md |
| 5 | 一致性檢查 | 見 Step 5（任意階段可用） | 檢查報告 |

---

### Step 1：產出功能規格

**第一次產出**時，複製以下內容貼給 Claude：

```
請依據 requirements/RQ{模組代碼}.md 和 use-cases/{模組代碼}/usecases.md，
產出 specs/{模組代碼}/spec.md 功能規格。

來源檔案：
- 需求文件：requirements/RQ{模組代碼}.md
- 使用案例：use-cases/{模組代碼}/usecases.md

產出要求：
1. 請依循 CLAUDE.md 規範
```

**範例（供應模組）**：

```
請依據 requirements/RQBS.md 和 use-cases/bs/usecases.md，
產出 specs/bs/spec.md 功能規格。

來源檔案：
- 需求文件：requirements/RQBS.md
- 使用案例：use-cases/bs/usecases.md

產出要求：
1. 請依循 CLAUDE.md 規範
```

**更新既有規格**時：

```
請依循 CLAUDE.md 規範，依據 requirements/RQ{模組代碼}.md 和 use-cases/{模組代碼}/usecases.md，
重新產出 specs/{模組代碼}/spec.md。
```

---

### Step 2：釐清規格

spec.md 產出後，請 Claude 檢查有沒有不明確的地方：

```
請依循 CLAUDE.md 規範，檢查 specs/{模組代碼}/spec.md，找出不明確或待確認的地方，提出釐清問題。
```

**注意**：
- Claude 每次最多提出 5 個問題，回答後會更新 spec.md
- **可多輪執行**，直到確認沒有需要釐清的問題後才進入下一步
- 回答方式：直接回覆答案即可，例如「Q1: A, Q2: 用預設值, Q3: 不需要」

---

### Step 3：產出實作計畫

spec.md 確認後，請 Claude 產出實作計畫：

```
請依循 CLAUDE.md 規範，依據 specs/{模組代碼}/spec.md，產出實作計畫。
```

Claude 會產出以下檔案：
- `plan.md` — 實作計畫（技術背景、開發順序、跨模組依賴）
- `research.md` — 研究決策紀錄（設計選擇與排除方案）
- `data-model.md` — 資料模型（ERD + DD + 代碼表）
- `contracts/` — 介面契約（有外部 API 時才產出）

---

### Step 4：產出開發任務

plan.md 確認後，請 Claude 產出開發任務清單：

```
請依循 CLAUDE.md 規範，依據 specs/{模組代碼}/ 下的設計文件，產出開發任務清單。
```

Claude 會產出 `tasks.md`，任務依 User Story 分群。

---

### Step 5：一致性檢查

任意階段都可以請 Claude 做跨文件一致性檢查：

```
請依循 CLAUDE.md 規範，檢查 specs/{模組代碼}/ 下所有文件的一致性，
確認 spec.md、plan.md、data-model.md、tasks.md 之間有無矛盾。
```

---

### Step 6：開立 GitHub Issues

tasks.md 確認後，可依功能作業開立 GitHub Issues。

**前置準備**：先產出 `specs/{模組代碼}/issues.md`（Issue 定義檔），確認內容無誤後再開立。

#### 6a. 產出 Issue 定義檔

```
請依據 specs/{模組代碼}/plan.md 的功能分群與 tasks.md 的任務清單，
產出 specs/{模組代碼}/issues.md。

規則：
1. 每張 Issue 對應一個功能畫面（垂直切割：含 DB + API + UI + 驗收條件）
2. 每張 Issue 包含：對應規格（US/FR/畫面）、涵蓋 Tasks、驗收條件、前置條件、Labels
3. 階段標註依 plan.md（P1-核心 / P2-延伸 / P3-輔助）
4. 報表類畫面各自獨立一張 Issue
5. Issue #0 為「專案建置與基礎建設」，涵蓋 Setup + Foundational 的所有 Tasks
```

#### 6b. 開立 GitHub Issues

確認 issues.md 無誤後，請 Claude 實際開立：

```
請依據 specs/{模組代碼}/issues.md，使用 gh 指令在 GitHub 上開立所有 Issues。

要求：
1. Issue 標題格式：[{階段}] {模組代碼} — {功能名稱}
2. Labels：{階段標籤}, {模組代碼}-{模組名}, {User Story}
3. 依序開立，在 body 中標註相依的 Issue 編號
```

**範例（供應模組）**：

```
請依據 specs/bs/issues.md，使用 gh 指令在 GitHub 上開立所有 Issues。

要求：
1. Issue 標題格式：[P1] BS — 層架管理
2. Labels：P1-核心, BS-供應, US1
3. 依序開立，在 body 中標註相依的 Issue 編號
```

#### Issue 拆分原則

| 原則 | 說明 |
|------|------|
| 一畫面一 Issue | 每個功能畫面（BS##）獨立一張 |
| 垂直切割 | 每張 Issue 包含該功能所需的 DB + API + UI |
| 報表各自獨立 | 報表不合併，各自一張方便指派 |
| 前置條件獨立 | 專案建置 + 基礎建設合併為 Issue #0 |
| 收尾可選 | 加密、效能、整合驗收視需要開立 |
